# Cloud Migration Plan

## Overview

Migrate persistence and API serving to Azure while keeping data collection local. The on-premises
`winet-integration` service continues polling the Sungrow WiNet inverter but no longer owns a
database. All data is forwarded to Azure Functions backed by Azure SQL Database (SQL Server).

### Why Connect Protocol over raw gRPC

gRPC requires HTTP/2 with specific framing that Azure Functions custom handlers do not support
natively (no additional infrastructure needed, lower cost). [Connect Protocol](https://connectrpc.com)
is the buf.build-native RPC protocol that:

- Is fully generated by `buf` with `protoc-gen-connect-go`
- Is gRPC-compatible (a gRPC client can call a Connect server)
- Works over plain HTTPS / HTTP/1.1 and HTTP/2
- Runs natively behind Azure Functions custom handler HTTP proxy with no additional infrastructure

If full gRPC framing is required in future, an Azure API Management layer can be placed in front
with no code changes to the function handlers.

### Why Azure Functions Custom Handler (Go)

Azure Functions does not have a native Go runtime. The custom handler model runs the Go binary
as an HTTP server that the Azure Functions host proxies requests to. This means:

- The Connect handler wiring is identical to the Lambda adapter pattern
- `host.json` configures the custom handler port and binary name
- No Azure SDK dependency needed in the handler code itself

## Phase 1: Proto + buf Setup

Define the shared contract between the local service and Azure Functions.

**`proto/winet/v1/ingestion.proto`** вҖ” key RPCs:

```protobuf
service IngestionService {
  rpc IngestData(IngestDataRequest) returns (IngestDataResponse);
  rpc RegisterDevice(RegisterDeviceRequest) returns (RegisterDeviceResponse);
  rpc IngestAmberPrices(IngestAmberPricesRequest) returns (IngestAmberPricesResponse);
  rpc IngestAmberUsage(IngestAmberUsageRequest) returns (IngestAmberUsageResponse);
}
```

**`proto/winet/v1/commands.proto`** вҖ” local service polls for pending inverter commands:

```protobuf
service CommandService {
  // Local service calls this on each poll cycle to check for pending commands
  rpc GetPendingCommands(GetPendingCommandsRequest) returns (GetPendingCommandsResponse);
  // Local service confirms a command was executed
  rpc AckCommand(AckCommandRequest) returns (AckCommandResponse);
}
```

**`buf.yaml`**: single module at the repo root, uses `buf.build/googleapis/googleapis` for
well-known types.

**`buf.gen.yaml`**: generates `go`, `connect-go` plugins into `gen/`.

## Phase 2: Local Service Changes

**Remove:**
- Direct database connection from `cmd/cmd.go`
- `setupDatabase` call and all `database.Database` usage
- `server` package startup (HTTP server + auth moves to `functions/api`)
- MQTT publisher (optional вҖ” keep if still useful for Home Assistant)

**Add:**
- `internal/pkg/grpcclient/` вҖ” Connect client wrapping the generated stub, with retry/backoff
- New `Publisher` implementation (`GRPCPublisher`) implementing `publisher.Publisher` interface,
  forwarding `Write` and `RegisterDevice` calls to the ingestion function
- Amber cron jobs forward data via gRPC instead of writing to DB
- Config: `FUNCTION_INGESTION_URL`, `FUNCTION_API_KEY`

**Feed-in controller** stays local вҖ” it reacts to device status changes in-memory and issues
inverter commands directly over the WiNet socket. No cloud round-trip needed for control.

**Amber price callback** (`feedinCtrl.Evaluate`) вҖ” prices arrive via gRPC response after ingestion;
the function can return the latest prices in the `IngestAmberPricesResponse` so the local service
can evaluate feed-in logic without needing to query back.

## Phase 3: Function вҖ” Ingestion Service (`functions/ingestion`)

A Go binary built for `linux/amd64`, deployed as an Azure Functions custom handler.

**Responsibilities:**
- Implement `IngestionService` and `CommandService` Connect handlers
- Write received data to Azure SQL using `sqlc`-generated queries (SQL Server dialect)
- Run DB migrations on startup or as a separate step in CI (preferred)
- Expose via Azure Functions HTTP trigger (no API Management needed for server-to-server)

**Custom handler wiring:**
```go
// Azure Functions custom handler: Go binary listens on the port provided by FUNCTIONS_CUSTOMHANDLER_PORT
port := os.Getenv("FUNCTIONS_CUSTOMHANDLER_PORT")
mux := http.NewServeMux()
mux.Handle(ingestionv1connect.NewIngestionServiceHandler(svc, authInterceptor))
mux.Handle(commandv1connect.NewCommandServiceHandler(svc, authInterceptor))
http.ListenAndServe(":"+port, mux)
```

**`host.json`:**
```json
{
  "version": "2.0",
  "customHandler": {
    "description": {
      "defaultExecutablePath": "ingestion",
      "workingDirectory": "",
      "arguments": []
    },
    "enableForwardingHttpRequest": true
  }
}
```

`enableForwardingHttpRequest: true` passes the raw HTTP request directly to the Go handler,
which is required for Connect Protocol to work correctly.

## Phase 4: Function вҖ” API Service (`functions/api`)

Serves the existing OpenAPI-defined HTTP REST endpoints to frontends.

- Move `internal/pkg/server/` and `internal/pkg/auth/` into `functions/api/internal/`
- Connect to same Azure SQL instance
- JWT auth unchanged
- Azure Functions HTTP trigger (auth level: anonymous вҖ” JWT handles it)

**WinetService commands** вҖ” the frontend posts battery/inverter commands. These are queued in
a `pending_commands` table in Azure SQL. The local service polls `GetPendingCommands` and executes
them, then acks back. This decouples the frontend from the local service entirely.

---

## Phase 5: Server-to-Server Authentication

The local service вҶ’ ingestion function channel must be locked down.

**Mechanism: Shared API key (Bearer token)**

1. Generate a strong random key (e.g., 32 bytes, base64-encoded)
2. Store it as:
   - An Azure Function App Setting: `INGESTION_API_KEY`
   - Local service env var: `FUNCTION_API_KEY`
3. Local service sends `Authorization: Bearer <key>` on every Connect request
4. Ingestion function validates the key in a Connect interceptor before any handler runs

**Why not mTLS?**
mTLS is more robust but requires certificate management. A rotatable pre-shared key is simpler
to operate for a private home system and can be upgraded later.

**Frontend вҶ’ API Function:**
Unchanged вҖ” existing JWT flow (login вҶ’ access token + refresh cookie).

## Phase 6: Azure Infrastructure

All infrastructure is set up manually (no Terraform in scope). Required resources:

| Resource | Notes |
|---|---|
| Azure SQL Database (free tier) | Always-free tier: 32 GB, serverless, SQL Server dialect. Same schema as existing (migrated from Postgres). |
| Azure Function App: `winet-ingestion` | Consumption plan (free tier). Custom handler, `linux/amd64`. App Settings: `DATABASE_URL`, `INGESTION_API_KEY`. |
| Azure Function App: `winet-api` | Consumption plan (free tier). Custom handler, same resource group. App Settings: `DATABASE_URL`, `JWT_SECRET`, `ALLOWED_ORIGIN`. |
| Azure Storage Account | Required by Functions runtime for deployment artifacts and state. |
| Resource Group | Single resource group for all above resources. |

**DB Migrations**: Run via `golang-migrate` CLI in the GitHub Actions deploy pipeline against
Azure SQL before the function code is updated. Azure SQL supports `sqlserver` driver.

**sqlc dialect**: Update `sqlc.yaml` to use `sqlserver` engine. Queries use T-SQL syntax
(e.g., `IDENTITY` instead of `SERIAL`, `NVARCHAR` instead of `TEXT`).

## Phase 7: GitHub Actions вҖ” Deploy Pipeline

**Trigger:** push to `main` (or manual `workflow_dispatch`)

**Steps:**
1. `buf generate` вҖ” regenerate protos (fails build if out of sync)
2. `go test ./...` вҖ” run all tests
3. Build ingestion function binary: `GOOS=linux GOARCH=amd64 go build -o ingestion ./functions/ingestion`
4. Build api function binary: `GOOS=linux GOARCH=amd64 go build -o api ./functions/api`
5. Package each function app (binary + `host.json` + `function.json`) into a zip
6. Run DB migrations against Azure SQL via `golang-migrate`
7. `az functionapp deployment source config-zip` for each function app

**Required GitHub Actions secrets (set manually):**
- `AZURE_CREDENTIALS` вҖ” service principal JSON (`az ad sp create-for-rbac --sdk-auth`)
- `DATABASE_URL` вҖ” Azure SQL connection string (for migration step)
- `AZURE_INGESTION_APP_NAME` вҖ” Function App name
- `AZURE_API_APP_NAME` вҖ” Function App name
- `AZURE_RESOURCE_GROUP` вҖ” Resource group name

## Schema Migration: Postgres вҶ’ SQL Server

The existing schema uses Postgres-specific syntax that must be converted:

| Postgres | SQL Server equivalent |
|---|---|
| `SERIAL` / `BIGSERIAL` | `INT IDENTITY(1,1)` / `BIGINT IDENTITY(1,1)` |
| `TEXT` | `NVARCHAR(MAX)` |
| `BOOLEAN` | `BIT` |
| `NOW()` | `GETUTCDATE()` |
| `RETURNING id` | `OUTPUT INSERTED.id` |
| `ON CONFLICT DO UPDATE` | `MERGE` statement |

All migration files under `migrations/` must be rewritten in T-SQL. `golang-migrate` with
`sqlserver` driver handles the migration table automatically.

## Open Questions / Decisions

| # | Question | Recommendation |
|---|---|---|
| 1 | Keep MQTT publisher on local service? | Yes, keep it вҖ” zero cost, useful for Home Assistant |
| 2 | Amber polling: keep local or move to Function timer trigger? | Keep local for now (simpler); Azure timer trigger is a future improvement |
| 3 | Commands: queue in DB or in-memory cache in function? | DB table (`pending_commands`) вҖ” survives function cold starts |
| 4 | Single Function App with route-based dispatch vs two separate apps? | Two separate вҖ” clean separation of concerns, independent scaling + deploy |
| 5 | API key rotation strategy? | Manual rotation via App Settings update + redeploy; automate with Key Vault reference later |
| 6 | sqlc: rewrite queries in T-SQL or use a DB-agnostic layer? | Rewrite in T-SQL вҖ” sqlc's sqlserver engine generates correct types |
| 7 | Azure SQL free tier limits? | 32 GB storage, 100,000 vCore-seconds/month вҖ” more than sufficient for home use |

## Implementation Order

1. **Proto + buf** вҖ” define messages and services, generate Go code
2. **Schema migration** вҖ” rewrite migrations in T-SQL for Azure SQL
3. **sqlc** вҖ” switch engine to `sqlserver`, regenerate queries
4. **gRPC publisher** вҖ” new `Publisher` implementation in local service
5. **Ingestion function** вҖ” Connect server + Azure SQL writes, custom handler wiring
6. **Local service wiring** вҖ” swap DB publisher for gRPC publisher, strip DB startup
7. **Commands table + polling** вҖ” DB schema + `GetPendingCommands` / `AckCommand`
8. **API function** вҖ” move HTTP server + auth, wire to Azure SQL
9. **GitHub Actions deploy** вҖ” CI/CD pipeline
10. **Cut over** вҖ” point local service at function URL, verify data flow end-to-end
