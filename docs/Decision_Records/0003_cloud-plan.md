# Cloud Migration Plan

## Overview

Migrate persistence and API serving to AWS while keeping data collection local. The on-premises
`winet-integration` service continues polling the Sungrow WiNet inverter but no longer owns a
database. All data is forwarded to AWS Lambda functions backed by RDS Postgres.

### Why Connect Protocol over raw gRPC

gRPC requires HTTP/2 with specific framing that AWS Lambda Function URLs do not support natively
(no ALB needed, lower cost). [Connect Protocol](https://connectrpc.com) is the buf.build-native
RPC protocol that:

- Is fully generated by `buf` with `protoc-gen-connect-go`
- Is gRPC-compatible (a gRPC client can call a Connect server)
- Works over plain HTTPS / HTTP/1.1 and HTTP/2
- Runs natively behind Lambda Function URLs with no additional infrastructure

If full gRPC framing is required in future, an ALB can be placed in front with no code changes
to the Lambda handlers.

## Phase 1: Proto + buf Setup

Define the shared contract between the local service and Lambda.

**`proto/winet/v1/ingestion.proto`** — key RPCs:

```protobuf
service IngestionService {
  rpc IngestData(IngestDataRequest) returns (IngestDataResponse);
  rpc RegisterDevice(RegisterDeviceRequest) returns (RegisterDeviceResponse);
  rpc IngestAmberPrices(IngestAmberPricesRequest) returns (IngestAmberPricesResponse);
  rpc IngestAmberUsage(IngestAmberUsageRequest) returns (IngestAmberUsageResponse);
}
```

**`proto/winet/v1/commands.proto`** — local service polls for pending inverter commands:

```protobuf
service CommandService {
  // Local service calls this on each poll cycle to check for pending commands
  rpc GetPendingCommands(GetPendingCommandsRequest) returns (GetPendingCommandsResponse);
  // Local service confirms a command was executed
  rpc AckCommand(AckCommandRequest) returns (AckCommandResponse);
}
```

**`buf.yaml`**: single module at the repo root, uses `buf.build/googleapis/googleapis` for
well-known types.

**`buf.gen.yaml`**: generates `go`, `connect-go` plugins into `gen/`.

## Phase 2: Local Service Changes

**Remove:**
- Direct Postgres database connection from `cmd/cmd.go`
- `setupDatabase` call and all `database.Database` usage
- `server` package startup (HTTP server + auth moves to `lambda/api`)
- MQTT publisher (optional — keep if still useful for Home Assistant)

**Add:**
- `internal/pkg/grpcclient/` — Connect client wrapping the generated stub, with retry/backoff
- New `Publisher` implementation (`GRPCPublisher`) implementing `publisher.Publisher` interface,
  forwarding `Write` and `RegisterDevice` calls to the ingestion Lambda
- Amber cron jobs forward data via gRPC instead of writing to DB
- Config: `LAMBDA_INGESTION_URL`, `LAMBDA_API_KEY`

**Feed-in controller** stays local — it reacts to device status changes in-memory and issues
inverter commands directly over the WiNet socket. No cloud round-trip needed for control.

**Amber price callback** (`feedinCtrl.Evaluate`) — prices arrive via gRPC response after ingestion;
Lambda can return the latest prices in the `IngestAmberPricesResponse` so the local service can
evaluate feed-in logic without needing to query back.

## Phase 3: Lambda — Ingestion Service (`lambda/ingestion`)

A single Go binary built for `linux/arm64`, deployed as a Lambda function using the `arm64`
architecture (`AL2023` runtime). ARM (Graviton2) is ~20% cheaper per GB-second than x86 and
sufficient for the low-compute ingestion workload.

**Responsibilities:**
- Implement `IngestionService` and `CommandService` Connect handlers
- Write received data to RDS Postgres using the existing `sqlc`-generated queries
- Run DB migrations on cold start (or separate migration step in CI — preferred)
- Expose a Lambda Function URL (no API Gateway needed for server-to-server)

**Handler wiring** (Lambda adaptor pattern):
```go
// Lambda receives HTTP events; Connect handlers work over plain HTTP
mux := http.NewServeMux()
mux.Handle(ingestionv1connect.NewIngestionServiceHandler(svc, authInterceptor))
mux.Handle(commandv1connect.NewCommandServiceHandler(svc, authInterceptor))
lambda.Start(httpadapter.NewV2(mux).ProxyWithContext)
```

Uses `github.com/aws/aws-lambda-go` + `github.com/awslabs/aws-lambda-go-api-proxy`.

## Phase 4: Lambda — API Service (`lambda/api`)

Serves the existing OpenAPI-defined HTTP REST endpoints to frontends.

- Move `internal/pkg/server/` and `internal/pkg/auth/` into `lambda/api/internal/`
- Connect to same RDS instance
- JWT auth unchanged
- Lambda Function URL (with IAM auth disabled — JWT handles it)

**WinetService commands** — the frontend posts battery/inverter commands. These are queued in
a `pending_commands` table in RDS. The local service polls `GetPendingCommands` and executes them,
then acks back. This decouples the frontend from the local service entirely.

## Phase 5: Server-to-Server Authentication

The local service → ingestion Lambda channel must be locked down.

**Mechanism: Shared API key (Bearer token)**

1. Generate a strong random key (e.g., 32 bytes, base64-encoded)
2. Store it as:
   - A Lambda environment variable: `INGESTION_API_KEY` (set via GitHub Actions secret or manually)
   - Local service env var: `LAMBDA_API_KEY`
3. Local service sends `Authorization: Bearer <key>` on every Connect request
4. Ingestion Lambda validates the key in a Connect interceptor before any handler runs

**Why not mTLS?**
mTLS is more robust but requires certificate management. A rotatable pre-shared key is simpler
to operate for a private home system and can be upgraded later.

**Frontend → API Lambda:**
Unchanged — existing JWT flow (login → access token + refresh cookie).

**API Lambda → Ingestion Lambda (if needed):**
Same API key mechanism via an internal HTTP call; or direct RDS access (preferred, avoids a
hop).

## Phase 6: AWS Infrastructure

All infrastructure is set up manually (no Terraform in scope). Required resources:

| Resource | Notes |
|---|---|
| RDS Postgres (t3.micro or t4g.micro) | Free tier eligible. Same schema as existing. Enable automated backups. |
| Lambda: `winet-ingestion` | 128 MB, 30s timeout, `arm64` architecture, VPC subnet with RDS access |
| Lambda: `winet-api` | 128 MB, 15s timeout, `arm64` architecture, same VPC |
| Lambda Function URLs | Both lambdas get public HTTPS URLs (auth via API key / JWT) |
| Secrets Manager or env vars | `DATABASE_URL`, `INGESTION_API_KEY`, `JWT_SECRET` |
| VPC + Security Groups | Lambda SG → RDS SG on 5432 only |
| ECR or S3 | Lambda deployment artifact (zip or container image) |

**DB Migrations**: Run as a one-off Lambda invocation or a separate `migrate` Lambda triggered
in the GitHub Actions deploy pipeline before the main Lambda is updated.

---

## Phase 7: GitHub Actions — Deploy Pipeline

**Trigger:** push to `main` (or manual `workflow_dispatch`)

**Steps:**
1. `buf generate` — regenerate protos (fails build if out of sync)
2. `go test ./...` — run all tests
3. Build ingestion Lambda binary: `GOOS=linux GOARCH=arm64 go build ./lambda/ingestion`
4. Build api Lambda binary: `GOOS=linux GOARCH=arm64 go build ./lambda/api`
5. Zip each binary
6. Run DB migrations (invoke migrate Lambda or use `golang-migrate` CLI against RDS via bastion/SSM)
7. `aws lambda update-function-code` for each function

**Required GitHub Actions secrets (set manually):**
- `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` — IAM user with Lambda update + S3 permissions
- `DATABASE_URL` — RDS connection string (for migration step)
- `LAMBDA_INGESTION_ARN` — Lambda function ARN
- `LAMBDA_API_ARN` — Lambda function ARN

## Open Questions / Decisions

| # | Question | Recommendation |
|---|---|---|
| 1 | Keep MQTT publisher on local service? | Yes, keep it — zero cost, useful for Home Assistant |
| 2 | Amber polling: keep local or move to Lambda cron (EventBridge)? | Keep local for now (simpler); Lambda EventBridge trigger is a future improvement |
| 3 | Commands: queue in DB or in-memory cache in Lambda? | DB table (`pending_commands`) — survives Lambda cold starts |
| 4 | Lambda container image vs zip deployment? | Zip is simpler to start; switch to container if binary > 50 MB |
| 5 | Single Lambda with URL routing vs two separate Lambdas? | Two separate — clean separation of concerns, independent scaling + deploy |
| 6 | API key rotation strategy? | Manual rotation via env var update + redeploy; automate with Secrets Manager later |
| 7 | RDS in public subnet (with SG) vs private subnet + NAT? | Private subnet + Lambda in same VPC — no NAT needed for Lambda→RDS |

## Implementation Order

1. **Proto + buf** — define messages and services, generate Go code
2. **gRPC publisher** — new `Publisher` implementation in local service
3. **Ingestion Lambda** — Connect server + RDS writes
4. **Local service wiring** — swap DB publisher for gRPC publisher, strip DB startup
5. **Commands table + polling** — DB schema + `GetPendingCommands` / `AckCommand`
6. **API Lambda** — move HTTP server + auth, wire to RDS
7. **GitHub Actions deploy** — CI/CD pipeline
8. **Cut over** — point local service at Lambda URL, verify data flow end-to-end
